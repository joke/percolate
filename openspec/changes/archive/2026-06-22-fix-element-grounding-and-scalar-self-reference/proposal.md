## Why

Two paradigm-agnostic mappers that *should* generate today report a clean `no plan`, both parked as follow-ups by the just-archived `fix-container-return-mappers` / `add-reactor-modules` work: (1) a **scalar self-referential** mapper (`HumanNode mapNode(PersonNode)` walking a `next` chain) — the self-bridge fix excludes the method-under-generation from its *entire* scope, which also kills the legitimate `this.mapNode(src.getNext())`; and (2) a **reactive element transform** (`Flux<DTO> → List<DAO>` delegating elements to `this::mapOne`) — the JDK element-map cannot ground its element type from a reactive source. Both are "this was supposed to work" gaps in shipped capabilities, and both are isolated; they ship together.

## What Changes

- **FU1 — argument-aware self-call rule (per-binding, not per-scope).** The self-call exclusion is genuinely a *binding* decision: at one target site the engine over-emits both `this.m(src)` (degenerate, arg = the whole param) and `this.m(src.getNext())` (legitimate, arg = a sub-part), and the degenerate one is *strictly cheaper* (param-root costs 0, an accessor costs `ACCESS`), so over-emit + cost-prune cannot save it. Replace today's scope-wide `SelfCallGuard` visibility filter with a bind-time prohibition: **a method may not be called on its own whole parameter; a sub-part of the parameter is allowed.** This needs the driver to know, at bind time, *which method an operation calls* — carried as a **neutral, additive** call-target fact on `OperationSpec` (the `ExecutableElement` identity `MethodCallBridge` already holds), *not* a "self-call marker" (a marker would require a strategy to know the method-under-generation, which it deliberately cannot). The driver (which owns the `MethodScope`) compares the call target to the scope's method and refuses only the whole-parameter binding.
- **FU2 — opt-in blocking `SourceProjection` for reactive → JDK grounding.** The `reactor-blocking` module shipped the *producers* (`FluxToStream`, `FluxCollectListBlock`) but no `SourceProjection`, so a `Flux<X>` source contributes only `Flux<X>` to the grounding match set and the JDK `StreamMap`'s `Stream<A>` port cannot unify — the element type is ungroundable and no element map is emitted. Add the matching **opt-in** grounding view `Flux<X> → Stream<X>` (cleanest: `FluxToStream` *also* implements `SourceProjection`, one class = view + producer, mirroring how `Container` bundles `project()` + `expand()`), plus `Mono<X> → Optional<X>` projected from the **total** `MonoBlockOptional`. The projection only enables the type-binding; the real intermediate is still produced by the existing high-weight (`Blockings.WEIGHT`) reuse-only bridge, so no eager block is invented and the boundary-direction rule stays enforced by *packaging*.
- **Correct the stale D4 note.** `add-reactor-modules`/`fix-container-return-mappers` framed this as a "cost-tuning follow-up" assuming a wrong-but-compiling `List.of(flux.single().block())` shortcut; that shortcut does not exist (no singleton-List-wrap producer). The honest state is `no plan`, and the true cause is the missing grounding view.

### Modified Capabilities
- `graph-expansion`: the self-call rule becomes per-binding — a method never binds its **own whole parameter** as a self-call argument, but a **sub-part** of the parameter (a `next`-style accessor result) is a legitimate self-call; legitimate container-element recursion (separate child scope) and delegation to a *different* method are unchanged.
- `expansion-strategy-spi`: `OperationSpec` carries an optional, neutral **call-target** identity (additive — existing `of`/`ofPartial`/`mapping` callers unchanged); a method-call strategy records the method it calls so the driver can apply the binding-time self-call rule without label inference.
- `reactor-containers`: the opt-in `reactor-blocking` module additionally registers blocking **`SourceProjection`s** (`Flux<X> → Stream<X>`, `Mono<X> → Optional<X>` from the total bridge) so a JDK element transform can ground its element type against a reactive source; an upward element-transform crossing (`Flux<DTO> → List<DAO>`) becomes satisfiable when, and only when, `reactor-blocking` is on the classpath.

## Impact

- **Engine (`processor`):** `OperationSpec`-consuming bind path in `ExpandStage` (driver) gains the whole-parameter self-call refusal; `SelfCallGuard` collapses its scope-wide visibility filter into a bind-time check; `MethodCallBridge:38-42`'s existing assembly-case half-guard unifies in; `AccessorResolver.resolveAccessor` swaps its `output != parentType` proxy for `!port.isReuseOnly()` so a recursive same-type accessor (`getNext():NodeDto`) resolves and the scalar self-reference generates end-to-end (see design D6).
- **SPI (`spi`):** `OperationSpec` gains an additive optional call-target field; `MethodCandidate.getMethod()` identity (already public) is now also surfaced on the output spec — symmetric, no new leak.
- **Builtins (`strategies-builtin`):** `MethodCallBridge` populates the new call-target fact.
- **`reactor-blocking` module:** `FluxToStream` (+ a `Mono → Optional` view) additionally implement `SourceProjection`; `@AutoService` updated. No engine/SPI dependency.
- **Tests:** new engine spec for the scalar self-reference positive + whole-param-still-`no plan` negative; `CatMapper`/different-method-delegation stay green; `reactor-blocking` positives for the param-direct `Flux<DTO> → List<DAO>` minimal repro, the bean-field `src.people` case, and `Mono<DTO> → Optional<DAO>`; the "no eager block" weight-inversion guard retained.
- **Known risk (needs integration `.dot`):** in the bean-field case the `Flux<DTO>` ACCESS source must be **materialised** in scope before the element-map grounding runs, or its type is absent from `sourceTypes`; may need a small materialisation-order tweak in `graph-expansion`. The param-direct repro isolates the projection fix from this. Compile-testing cannot dump `.dot` on a failing compile, so diagnosis goes through the `percolate-integration` build.
- **Non-goals:** do not make the over-emit graph acyclic (box/unbox cycles are intentional); no auto upward crossings (opt-in by packaging); strategies stay myopic, all graph mutation in driver/Applier; no forward source→target expansion.
- **Affected teams:** percolate engine + reactor plugin maintainers (single-maintainer project).
