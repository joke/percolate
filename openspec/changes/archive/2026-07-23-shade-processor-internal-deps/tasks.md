## 2. Drop jgrapht-io; shade jgrapht-core, auto-common, and dagger (plus their own runtime deps) into processor

- [x] 2.1 Replace `DotRenderer`'s `org.jgrapht.nio`/`org.jgrapht.nio.dot.DOTExporter` usage with a small hand-written DOT serializer (vertex/edge statement + attribute-map rendering); `DotRenderer` already built its own attribute maps and vertex-ID quoting, so `DOTExporter` was only responsible for final text assembly. Update `vertexAttributes`/`edgeAttributes` to return `Map<String, String>` (drop the `Attribute`/`DefaultAttribute` wrapper). Update `DotRendererSpec` accordingly.
- [x] 2.2 Add the `com.gradleup.shadow` plugin to `processor/build.gradle`. Add a standalone `shaded` configuration (not extended by `compileOnly`/`implementation`) so its merge set and exclude rules never affect `processor`'s own compileClasspath.
- [x] 2.3 Change `jgrapht-core`, `auto-common`, and `dagger` (not `jgrapht-io`, dropped) to be declared on both `compileOnly` (compilation visibility) and the standalone `shaded` configuration (versions from `platform(project(':dependencies'))` on each). Exclude `jakarta.inject`/`javax.inject` and the small compile-time-only annotation jars (`jsr305`, `checker-qual`, `error_prone_annotations`, `j2objc-annotations`, `listenablefuture`) from `shaded`'s own resolution.
- [x] 2.4 Add explicit-versioned `implementation 'jakarta.inject:jakarta.inject-api:2.0.1'` and `implementation 'javax.inject:javax.inject:1'` (kept unshaded — `dagger.internal.Provider` implements both JSR-330 `Provider` interfaces, needed unrelocated at runtime).
- [x] 2.5 Add a `shadowJar` task: `archiveClassifier.set('')`, `configurations.set([project.configurations.shaded])`, and `relocate` rules for `org.jgrapht`, `org.jheaps`, `org.apfloat` (jgrapht-core's own runtime deps), `com.google.auto.common`, `com.google.common` (Guava — needed since `BasicAnnotationProcessor` embeds it directly), and `dagger` (the library's actual top-level package, **not** its `com.google.dagger` Maven groupId — relocating the groupId string was a real bug caught during verification, since it silently matches zero classes).
- [x] 2.6 Override `apiElements`/`runtimeElements` outgoing artifacts to publish the `shadowJar` output, mirroring `percolate-javapoet`. Set `shadow { addShadowVariantIntoJavaComponent = false }`. **Also** set `configurations.shadowRuntimeElements.canBeConsumed = false` and `shadowApiElements.canBeConsumed = false` — without this, Gradle's variant matching prefers Shadow's own zero-dependency shadow variants over the real ones for ordinary in-build `project(':processor')` consumers, silently dropping `jakarta.inject`/`javax.inject`/`annotations`/`spi` for them (surfaced as `NoClassDefFoundError` in `reactor`/`reactor-blocking` integration tests, which run `PercolateProcessor` in-process).
- [x] 2.7 Move the `:dependencies` platform import from `implementation` to `compileOnly` in `processor/build.gradle` (nothing in `implementation` needs it anymore — `annotations`/`spi` are project deps, `jakarta.inject`/`javax.inject` have explicit versions), so it no longer leaks an unresolvable `import`-scope `dependencyManagement` entry into the published POM.
- [x] 2.8 Add `testImplementation 'org.jgrapht:jgrapht-core'`, `'com.google.auto:auto-common'`, `'com.google.dagger:dagger'` to `processor` — tests that exercise real jgrapht/dagger runtime behavior (e.g. `PercolateProcessorSpec` building a real Dagger component, `BipartiteGraphSpec`) need the real (unrelocated) jars, since shading only applies to the published `shadowJar`, not the ordinary test classpath.

## 3. Guard the shading invariants

- [x] 3.1 Considered a source-level ArchUnit guard (mirroring `JavaPoetRelocationSpec`) asserting no production import of the unrelocated shaded packages — **rejected**: unlike JavaPoet, `processor`'s own source is the shading input and necessarily imports the raw upstream types directly (relocation is a post-compile rewrite, not a source-level restriction); such a rule can never hold and was confirmed to fail immediately when tried.
- [x] 3.2 No jar-inspection "swallow" test added either, matching the established precedent from `decentralize-architecture-boundary-checks` (which deleted the equivalent JavaPoet jar-inspection check outright, accepting the risk for a narrow-blast-radius invariant). The invariant is verified manually (2.9/2.10) plus end-to-end via `percolate-smoke`.
- [x] 3.3 Fixed `ModuleBoundariesSpec`'s pre-existing `javax.lang.model.util` confinement rule, which started flagging the newly-relocated `io.github.joke.percolate.lib.autocommon.*` visitor classes: broadened its JavaPoet-only exclusion filter to the whole `/io/github/joke/percolate/lib/` prefix, covering every shaded library uniformly.
