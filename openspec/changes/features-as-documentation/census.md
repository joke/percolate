## 3.5 Locked doc-e2e pattern (spike, tasks 2.1-2.3)

Proven end-to-end on `getting-started`/`PersonMapper` — the template every later chapter conversion (4.x, 5.x, 6.x) reuses:

1. The example `@Mapper` fixture is **real source** under `<module>/src/test/java/io/github/joke/percolate/docs/<chapter>/*.java` (not a `src/test/resources` compile-testing fixture), with AsciiDoc `// tag::mapper[]`/`// tag::model[]` region markers as before. Needs a `package-info.java` with `@org.jspecify.annotations.NullMarked` in the same package (NullAway's `RequireExplicitNullMarking` applies to test source too).
2. The owning module's `build.gradle` adds `testAnnotationProcessor project(':percolate')` (the real starter — engine + that module's own builtin strategies, exactly as a consumer applies it) and `testImplementation project(':annotations')`, plus `-Apercolate.docTags=true` on `compileTestJava.options.compilerArgs`. This is an ordinary test→main edge (`:percolate` depends on the module's own main jar) — not a cycle, Gradle resolves it fine.
3. `compileTestJava` (ordinary Gradle compilation, no compile-testing) now runs the real `PercolateProcessor` on the fixture, producing a real `*Impl.class` **and** the doc-tagged `*Impl.java` under Gradle's normal `build/generated/sources/annotationProcessor/java/test/...` — no custom `materialise()` step needed.
4. A plain `@Tag('integration')` Spock spec in the same package just does `new FooMapperImpl()` and asserts behaviour directly — zero `compile-testing`/`Compilation` involved.
5. `docs/antora.yml` gets two `scan` entries per converted module: one for the real source dir (`into: modules/ROOT/examples`) and one for the generated-output dir (same `into`) — the collector reconstructs `example$<chapter>/Foo.java` and `example$<chapter>/FooImpl.java` paths by dropping the scan `dir` prefix, so page `include::` paths must match the chapter subfolder name (no hyphens — Java package names can't have them; pages use the no-hyphen form, e.g. `example$gettingstarted/...` not `example$getting-started/...`).
6. The root `build.gradle`'s `antora` task depends on the relevant `compileTestJava` tasks (cheap — no full integrationTest run needed for converted chapters) in addition to `:strategies-builtin:integrationTest` while any category is still on the legacy pattern.
7. **Confirmed (task 2.2):** the collector also cleanly imports a module-owned `.adoc` **page** via a `scan` entry with `into: modules/ROOT/pages` (verified empirically with a throwaway probe page, then removed) — no second Antora content source is needed for D3's page co-location; each module just needs its own `scan` entry once pages actually move there (task 6.1).

## 5. This pass's own discovery: two real engine bugs, found while building row #12's example

Building the Optionals chapter's `PreferencesMapper` surfaced two genuine correctness defects in already-shipped `spi`/`processor` code — not documentation gaps. Both are fixed, with adapted/new unit tests (see row 27 above for the fix summary):

- **Spurious NPE on `Optional` wrap.** `Container.produceMyKind()`'s `wrap` operation hardcoded its scalar-element port to `NON_NULL` for every container kind — correct for `Mono`/`Flux` (whose `just` genuinely rejects null) but wrong for `Optional` (`ofNullable` is explicitly null-safe). A `@Nullable`-declared source in scope forced `NullnessCrossing`'s guard to bind, turning `Optional.ofNullable(bio)` into a crashing `Optional.ofNullable(Objects.requireNonNull(bio, …))`.
- **Mutual-recursion via conversion-method mis-reuse.** Two abstract mapper methods shaped `Optional<String> → String`, differing only by a `@Nullable` on the return (invisible to the Java type system), were selected as each other's reusable conversion method by `MethodCallBridge`/`SelfCallGuard` — generating infinite runtime recursion instead of each independently emitting `.orElseThrow()`/`.orElse(null)`.

No new e2e was added for either fix beyond the documentation-driven `OptionalsDocExampleSpec` that discovered them, per this change's own non-goal (exhaustiveness is the unit + pitest layer's job) — both fixes are covered by adapted `ContainerSpec`/`OptionalContainerSpec`/`SelfCallGuardSpec` unit specs instead.
