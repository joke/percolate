## Why

Root `build.gradle`'s `subprojects { project -> ... }` block (~340 lines: java/groovy/lombok/pitest/jacoco/pmd/errorprone/publishing conventions) is the entire configuration surface for every module, and it is structurally incompatible with Gradle's Isolated Projects feature — confirmed empirically by running `./gradlew help -Dorg.gradle.unsafe.isolated-projects=true`, which reports 652 configuration-cache problems (64 unique), one for essentially every line of the block. Isolated Projects also mandates configuration cache be enabled and forbids disabling it, which collides with a second, independent, already-known blocker: `org.shipkit.shipkit-auto-version`'s `git describe --tags` call fails configuration-cache storage (reproduced live; open upstream issue shipkit/shipkit-auto-version#134, no fix, no maintainer response) — and this repository currently has zero git tags (no release has ever been cut), so the failure is not an edge case, it reproduces on every invocation. Adopting Isolated Projects now, while the recent `restructure-publishing-plugin-wiring` change already reshaped publishing config into an Isolated-Projects-friendly reactive `withPlugin` cascade, extends that same shape to the rest of the build and unblocks faster, more parallel, more cacheable configuration as the module count grows.

## What Changes

- **BREAKING** (internal build wiring only, no consumer-facing effect): root `build.gradle`'s `subprojects { }` block is deleted entirely. Its content (java/groovy/lombok/pitest/jacoco/pmd/errorprone-nullaway/publishing conventions) is redistributed into precompiled Groovy script convention plugins under `buildSrc/`, applied explicitly by each module's own `plugins { }` block — mirroring the primary/additional plugin convention already established by `restructure-publishing-plugin-wiring`, just at `buildSrc` plugin granularity instead of root-script granularity.
- `org.shipkit.shipkit-auto-version` is replaced by a small hand-rolled `providers.exec()`-based git-describe computation in `percolate.base-conventions` (each module computes its own version independently, config-cache/Isolated-Projects-safe by construction) — degrading gracefully (bare commit hash via `--always`) instead of hard-failing when no git tag exists, with an explicit `-SNAPSHOT` policy for the non-tagged-commit case. `com.palantir.git-version` was tried first and dropped: its `apply()` unconditionally reaches into the root project's `pluginManager` from every project it's applied to, an Isolated Projects violation with no workaround (see design.md D4).
- `org.gradle.configuration-cache=true` (currently commented out in `gradle.properties`, per a prior known shipkit-auto-version incompatibility) is re-enabled — verified: with the two fixes above, `./gradlew check` and full multi-project configuration (`./gradlew projects`) both succeed cleanly under configuration cache.
- **Scope narrowed during implementation**: `org.gradle.unsafe.isolated-projects=true` is verified working (zero problems) for the `buildSrc`/version-source restructuring in isolation, but is **not** enabled by default in `gradle.properties`. `info.solidsoft.pitest` (applied by `processor`, `spi`, `strategies-builtin`) has its own unconditional root-touch — `PitestPlugin.apply()` always reaches into the root project's `buildscript` via a legacy-migration guard with no way to disable it, confirmed by decompiling `1.19.0` (the latest available release) — which this repository's own build script cannot fix. Actually flipping the flag on is deferred to a follow-up change, gated on an upstream fix to that plugin or another workaround; the property is left in `gradle.properties`, commented out, with this reasoning documented inline.

## Capabilities

### New Capabilities
- `isolated-projects-build`: defines that the build's cross-module configuration lives in `buildSrc` convention plugins rather than root-script `subprojects{}`/`allprojects{}`, that configuration cache is enabled by default, and that the `buildSrc`/version-source portion is verified Isolated-Projects-clean — while the flag itself stays off by default pending the `info.solidsoft.pitest` blocker.

### Modified Capabilities
- `release-versioning`: the "Version derived from git tags" requirement changes from `org.shipkit.shipkit-auto-version` to a `buildSrc`-hosted `providers.exec()` computation, adds an explicit no-tag/non-release-commit fallback scenario (previously undefined/failing), and gains a configuration-cache-compatibility constraint the old mechanism didn't meet.

## Impact

- Affected files: root `build.gradle` (subprojects block removed, only antora/wrapper-level root config remains), every module's `build.gradle` (new `plugins { }` ids for each applicable convention plugin), new `buildSrc/` directory (its own `build.gradle` applying `groovy-gradle-plugin`, plus one precompiled script plugin per today's `withPlugin` cluster), `gradle.properties` (configuration-cache re-enabled; isolated-projects flag documented but left off, see above), `settings.gradle` (the `org.shipkit.shipkit-auto-version` plugin coordinate removed, no replacement needed).
- No change to published artifact coordinates, POM content, module dependency graph, or CI publish gating (`release-please` still decides and tags releases; only how the Gradle side reads that tag changes).
- Every module's `build.gradle` `plugins { }` block grows (each convention plugin id is now explicit) in exchange for root `build.gradle` shrinking to near just the `antora`/`wrapper` blocks.
- `./gradlew` invocations become configuration-cache-backed by default (currently opt-out'd via a commented-out property due to the shipkit-auto-version bug); expect faster repeat configuration once warm, and a build-logic change now invalidates the whole config-cache entry (an accepted, named trade-off of using `buildSrc`).
