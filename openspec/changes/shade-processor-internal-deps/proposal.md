## Why

`processor`'s published POM currently declares `implementation` dependencies on `org.jgrapht:jgrapht-core`, `org.jgrapht:jgrapht-io`, `com.google.auto:auto-common`, and `com.google.dagger:dagger` with **no version** — their versions live only in the internal, unpublished `:dependencies` platform (`platform(project(':dependencies'))`), which per `maven-central-publishing` MUST NOT be published itself. Maven Central's publish validation rejects the resulting POM outright ("Dependency version information is missing"), which is the proximate trigger. The deeper problem, same rationale as `javapoet-relocation`: these are internal-only libraries (`PercolateProcessor` even `extends com.google.auto.common.BasicAnnotationProcessor`, and `DaggerProcessorComponent` is generated and consumed entirely within `processor`), so leaving them as ordinary external dependencies also exposes percolate to a foreign version of any of them colliding on a consumer's shared annotationProcessor classpath. This was explicitly deferred as a follow-up in `relocate-javapoet-as-spi-api`'s design doc (Non-Goals: "Shading the Dagger runtime and jgrapht-core/jheaps into the processor jar — internal-only deps, a separate follow-up change").

## What Changes

- Shade and relocate `org.jgrapht:jgrapht-core`, `org.jgrapht:jgrapht-io`, `com.google.auto:auto-common`, and `com.google.dagger:dagger` (runtime, not `dagger-compiler`) into `processor`'s own published jar, using the `com.gradleup.shadow` plugin already used by `percolate-javapoet`.
- Unlike JavaPoet, these libraries are consumed exclusively inside `:processor` (confirmed: no other module declares or imports them) and never cross the spi↔processor boundary, so this shades directly within `processor`'s own build rather than introducing a new shared relocation module.
- Establish one project-wide relocation convention — `io.github.joke.percolate.lib.<lib>` — for every shaded third-party dependency, and **align the existing `percolate-javapoet` relocation to it**: `io.github.joke.percolate.javapoet` → `io.github.joke.percolate.lib.javapoet`. **BREAKING**: every production import of `io.github.joke.percolate.javapoet.*` across `spi`, `processor`, `strategies-builtin`, `reactor`, `reactor-blocking` moves to `io.github.joke.percolate.lib.javapoet.*`.
- Move the `percolate-javapoet` module from the repo root to `lib/javapoet` (Gradle project path `:percolate-javapoet` → `:lib:javapoet`), establishing `lib/` as the home for any relocation module the project needs — package convention and directory structure now match. **BREAKING**: the module's Gradle project name (and therefore its published artifactId) changes from `percolate-javapoet` to `javapoet`, matching the directory; the artifactId is intentionally **not** pinned to its old value, so the coordinate becomes `io.github.joke.percolate:javapoet`. Any third-party libraries shaded in the future that do need their own module (unlike jgrapht/auto-common/dagger here, which don't) belong under `lib/` too.
- `processor`'s published POM/Gradle metadata SHALL declare zero dependency on any of the four newly-shaded coordinates after the change.
- Dagger's compile-time code generation (`dagger-compiler`, `annotationProcessor` scope) stays unrelocated as a build-time-only tool; only the `dagger` runtime artifact and `processor`'s own compiled classes (including Dagger-generated component/factory classes, which are package-private and fully internal already) get merged and relocated together in the same `shadowJar` pass, so the relocated hierarchy is self-consistent and self-contained in the shaded jar.
- **BREAKING** (theoretical only, jgrapht/auto-common/dagger): any external code directly importing `org.jgrapht`, `com.google.auto.common`, or `com.google.dagger` types obtained transitively through `processor` would break. No such usage exists in-repo (verified) and these were never part of any documented public API.

## Capabilities

### New Capabilities
- `processor-dependency-shading`: defines which internal-only third-party libraries `processor` shades/relocates into its own published jar under `io.github.joke.percolate.lib.*`, and the invariant that none of them leak as external dependencies or unrelocated classes in the published artifact.

### Modified Capabilities
- `javapoet-relocation`: the relocated package moves from `io.github.joke.percolate.javapoet` to `io.github.joke.percolate.lib.javapoet` to align with the new project-wide `io.github.joke.percolate.lib.*` shading convention. All requirements are otherwise unchanged (still fully swallowed, still `javax.lang.model.*` left unrelocated, still the sole codegen surface for spi/processor/strategy modules).

## Impact

- `processor/build.gradle`: add `com.gradleup.shadow`, switch `jgrapht-core`/`jgrapht-io`/`auto-common`/`dagger` from `implementation` to `compileOnly` (shading input), add a `shadowJar` task with four `relocate(...)` calls targeting `io.github.joke.percolate.lib.*`, override `apiElements`/`runtimeElements` outgoing artifacts to publish the shaded jar (mirroring `percolate-javapoet`), and adjust PMD/error-prone excluded-paths for the (now processor-local) generated Dagger sources if needed.
- `settings.gradle`: `include 'percolate-javapoet'` → `include 'lib:javapoet'`; the directory moves from `percolate-javapoet/` to `lib/javapoet/`.
- `lib/javapoet/build.gradle` (formerly `percolate-javapoet/build.gradle`): change the `relocate 'com.palantir.javapoet', ...` target from `io.github.joke.percolate.javapoet` to `io.github.joke.percolate.lib.javapoet`; no artifactId override — it follows the new project name (`javapoet`).
- `spi/build.gradle`, `bom/build.gradle`: update `project(':percolate-javapoet')` references to `project(':lib:javapoet')`.
- `spi`, `processor`, `strategies-builtin`, `reactor`, `reactor-blocking`: mechanical import rewrite from `io.github.joke.percolate.javapoet.*` to `io.github.joke.percolate.lib.javapoet.*` (IDE move-package style, per the precedent in `relocate-javapoet-as-spi-api`).
- No change to `dependencies/build.gradle` (stays internal-only, unpublished) beyond removing the now-shaded coordinates from being resolved as ordinary dependencies of `processor` — the version pins may still live there for the `compileOnly`/`shadow` input declarations.
- `architecture-tests`: extend the existing no-leaked-upstream-package ArchUnit/classpath guard pattern (from `javapoet-relocation`) to cover the four newly-shaded packages against `processor`'s published artifact, and update the existing javapoet guard's expected package to `io.github.joke.percolate.lib.javapoet`.
- `percolate-smoke` / `./gradlew check`: must stay green, confirming annotation processing still works end-to-end through the shaded jar on a real annotationProcessor classpath.
