## Why

The `source-path-resolvers` change types every directive's source chain and routes each directive's bridging SEED edge from the typed source to the typed target leaf, but the realisation engine never consumes that pinning structurally. Two pre-existing engine gaps remain — both exposed (not caused) by source-path-resolvers, and both blocking the integration scenarios on `~/Projects/joke/percolate-integration/mappers`:

1. `ResolveTargetChainsPhase` allocates typed target slots from `ConstructorCall.buildFor(...)` but does not consume the directive-carrying SEED edge feeding the corresponding untyped target leaf. The directive's specific source (e.g., `src[person2.first]:String`) is never bound to the typed slot (`tgt[firstName]:String`). `ExpandGroupsPhase` then realises the slot via type-only candidate selection — DirectAssign happily picks whichever same-typed source sorts first, ignoring directive intent.
2. `ExpandGroupsPhase.registerElementSeedGroup` materialises the element-seed input/output nodes and the nested group declared by `SetMap` (and `ListMap`/`OptionalMap`)'s `BridgeStep.elementSeeds`, but never emits the REALISED edge linking the outer iterable input candidate to the element-seed input. The slot is therefore unreachable from any source-parameter-root, the driver falls through to bridges, `OptionalWrap` misfires and synthesises an unproducible `Person.Address` intermediate, and the chain dead-ends.

Both gaps live in the driver/scaffolding layer the `target-to-source-expansion` architecture sanctions for that work. The fix preserves the locked-in invariants (target→source direction, strategies stay myopic, instance-identity nodes, per-group greedy commit) and changes no SPI.

Once the engine consumes the typed source chain via directive-pinning, the `GetterRead`-as-`Bridge` built-in (the legacy any-getter-to-target-type fallback) becomes truly redundant: every directive's source-path typing is now done at seed time by `GetterPathResolver` and the directive-pinned scaffolding wires the result directly to the typed target slot. `source-path-resolvers` deferred its removal to "after integration is green"; this change makes integration green and removes `GetterRead` in the same step so the regression window is one commit, not two.

## What Changes

- Extend `ResolveTargetChainsPhase` to scaffold a directive-pinned 1-slot `ExpansionGroup` when allocating each typed target slot. For each newly-allocated typed slot, the phase SHALL inspect the SEED edges incoming to the corresponding untyped target leaf; when exactly one such SEED edge originates at a typed source node (the `source-path-resolvers` typed bridging edge), the phase SHALL allocate a 1-slot `ExpansionGroup` whose root is the typed slot, whose single slot is that typed source node, whose codegen is a pass-through `(vars, inputs) -> CodeBlock.of("$L", inputs.single())`, and whose `strategyClassFqn` is a stable sentinel (`io.github.joke.percolate.processor.stages.expand.DirectiveBinding`). The group's initial REALISED edge runs slot → root. `MapperGraph.addGroup(group)` registers it.
- Extend `ExpandGroupsPhase.registerElementSeedGroup` to additionally emit a REALISED edge from the parent step's input candidate to the element-seed input node, alongside the nested element group it already registers. The edge carries the parent bridge's weight, a pass-through `EdgeCodegen` (codegen is owned by the parent SetMap/ListMap/OptionalMap step), and `strategyClassFqn` of the parent bridge.
- Move `registerElementSeedGroup`'s caller to pass the parent step's input candidate (it already has it locally in `commitBridgeStep`). No SPI change to `Bridge`, `BridgeStep`, or `ElementSeed`.
- **BREAKING (internal built-in only)**: delete `GetterRead`-as-`Bridge` and its `@AutoService(Bridge.class)` registration. Drop `GetterReadSpec`, `GetterReadMultiHopSpec`, and the `GetterRead` assertion in `BuiltinServiceRegistrationSpec`. Source-path getter access is owned by `GetterPathResolver` (seed-time, via `source-path-resolvers`); no engine-time fallback remains for "find a getter that returns this type."
- Add unit specs covering both scaffolding paths and update the integration acceptance to verify `tgt[firstName]` reads `person2.getFirst()`, `tgt[lastName]` reads `person.getLastName()`, and `tgt[addresses]` realises through the SetMap → ElementSeed → MethodCallBridge(mapAddress) chain at element scope. Verify also that the `GetterRead` service file no longer ships with the built-in JAR.

## Capabilities

### New Capabilities

None. Both engine fixes extend existing scaffolding/driver behaviour.

### Modified Capabilities

- `graph-expansion`: adds the directive-pinned target slot scaffolding requirement on `ResolveTargetChainsPhase`, and the element-seed iteration edge requirement on `ExpandGroupsPhase.registerElementSeedGroup`.
- `expansion-strategy-spi`: removes the `GetterRead` built-in from the listed registered strategies. The stale `GetterRead-as-SourceStep` requirement (left over from before `target-to-source-expansion`) is dropped entirely.
- `builtin-strategy-unit-tests`: drops `GetterReadSpec` from the required-specs list; the count of required built-in specs falls by one.
- `graph-debug-output`: drops the `GetterRead`-label scenarios from the renderer's strategy-FQN-to-label requirements.

## Impact

- **Affected code**: `processor/src/main/java/io/github/joke/percolate/processor/stages/expand/ResolveTargetChainsPhase.java`, `processor/src/main/java/io/github/joke/percolate/processor/stages/expand/ExpandGroupsPhase.java` (engine fixes). `strategies-builtin/src/main/java/io/github/joke/percolate/spi/builtins/GetterRead.java` deleted; `strategies-builtin/src/test/groovy/.../{GetterReadSpec,GetterReadMultiHopSpec}.groovy` deleted; `BuiltinServiceRegistrationSpec` amended. No SPI changes; no SeedGraph changes; no changes to remaining built-in strategies.
- **Affected behaviour**: `*.full.dot` now shows a `DirectiveBinding` cluster per resolved directive (typed source → typed slot), and a structural REALISED edge from each container iterable to its element-seed input. `*.transforms.dot` shows the directive's typed source directly feeding the typed target slot (no DirectAssign-via-other-candidate), and the SetMap chain reaches `mapAddress` through the element-scope OptionalUnwrap+MethodCallBridge chain. No edge in any generated dot will ever carry `strategyClassFqn == "io.github.joke.percolate.spi.builtins.GetterRead"` anymore.
- **Integration acceptance**: `~/Projects/joke/percolate-integration/mappers` with `mapAddress` present compiles; generated `PersonMapper.java` emits `person2.getFirst()` for `firstName`, `person.getLastName()` for `lastName`, and a `SetMap`-driven element loop calling `mapAddress` for `addresses`. With `mapAddress` commented out, the existing closest-miss diagnostic still names `Person.Address` as the deepest unresolved frontier.
- **Affected teams**: processor engine maintainers. Strategy authors writing their own `Bridge` implementations are unaffected — the SPI surface is unchanged. Downstream users who depended on the implicit "any-getter-to-target-type" bridging without an explicit `@Map` directive lose that behaviour; the supported equivalent is to write the directive explicitly, or wait for the future auto-directive-synthesis change.
- **Risk**: low. The engine fixes localise to driver-internal allocator helpers and add structural edges that are derivable from already-declared state. The `GetterRead` removal is mechanical once the engine consumes typed source pinning — every legitimate use-case in the integration mapper is covered by `GetterPathResolver` + directive-pinning. Existing realised-subgraph tests should continue to hold; the directive-binding group's pass-through edge is invisible to existing transforms-view assertions that test typed-slot incoming-edge counts, but those tests will need to recognise one additional REALISED in-edge per directive-pinned slot. Any test that constructs `GetterRead` directly is deleted.
