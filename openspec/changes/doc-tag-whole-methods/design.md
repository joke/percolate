## Context

The `docTags` codegen option ([[add-temporal-type-mapping]]'s directive-option rail) brackets generated
methods with AsciiDoc include-tag comments so the Antora manual single-sources real generated output. Today
`BuildMethodBodies.docTagged()` calls `DocTags.wrap(body, methodName)` — and `body` is the `CodeBlock`
JavaPoet drops *between* the braces. So the markers land inside the braces at body indentation:

```java
  public LocalDateTime toLocalDateTime(Instant timestamp) {
    // tag::toLocalDateTime[]
    return timestamp.atZone(ZoneId.systemDefault()).toLocalDateTime();
    // end::toLocalDateTime[]
  }
```

An `include::…Impl.java[tag=toLocalDateTime]` therefore extracts only the return statement, carrying its
in-brace indent, with no signature. This is **systemic** — every generated-impl `[tag=…]` snippet on every
manual page (`ProductMapperImpl[tag=map]`, `TeamMapperImpl[tag=toRoster]`, …) is headless and mis-indented.
The temporal page adds a second defect: line 95's `include::…TemporalMapperImpl.java[]` is a full-file
include with no tag filter, so AsciiDoc reproduces the `// tag::`/`// end::` marker lines verbatim.

AsciiDoc strips the tag directive lines and renders only what is *between* them, so showing the whole method
requires the markers to sit **outside** the braces (before the annotations, after the closing brace).
JavaPoet's `MethodSpec.emit()` is package-private and owns that whole span; a body `CodeBlock` cannot reach
it. Constraint: with `docTags` **off** (the default, every consumer), generated bytecode/source must stay
byte-for-byte identical ([[code-generation]] invariant).

`relocate-javapoet-as-spi-api` (2026-07-12) shades `com.palantir.javapoet:javapoet:0.16.0` into
`io.github.joke.percolate.javapoet` in the source-free `percolate-javapoet` module — which is exactly the
seam that makes patching a JavaPoet class viable now.

## Goals / Non-Goals

**Goals:**
- The `docTags` region brackets the **whole method** (signature through closing brace); a documentation
  `include::[tag=<method>]` renders a complete, correctly-indented method.
- The temporal whole-class listing renders the class with no `// tag::` marker noise.
- Consumer output with `docTags` off remains byte-for-byte identical.
- Keep codegen **structured** — no text post-processing of generated Java.

**Non-Goals:**
- Forking JavaPoet wholesale or vendoring its source tree — only a single class is overlaid.
- Changing the `docTags` option surface (`processor-options` is untouched) or any consumer-facing API.
- Re-authoring manual prose; only the `include::` attributes and the codegen shape change.
- Upgrading JavaPoet off the pinned `0.16.0`.

## Decisions

### D1 — Whole-method bracketing, not body-level

Move the tag region from the method body to the whole method. This is the actual spec change; everything
else is mechanism. The rendered per-method snippet becomes the full method:

```java
@Override
public LocalDateTime toLocalDateTime(Instant timestamp) {
  return timestamp.atZone(ZoneId.systemDefault()).toLocalDateTime();
}
```

### D2 — Overlay `MethodSpec` in `percolate-javapoet`, not text post-processing

| Option | Verdict |
|---|---|
| **Overlay `MethodSpec`** (chosen): a relocated source overlay adds leading/trailing tag emission in `emit()` | Structurally exact — JavaPoet still owns brace matching; `// end::` lands after the *real* closing brace by construction |
| Post-process the rendered source string (brace-scan per method) | Rejected: a brace-depth scan over generated Java is fooled by braces in string literals and **lambdas** — which the reactor strategies genuinely emit. Fragile exactly where output is richest |
| Abuse `MethodSpec.javadoc` to carry a `// tag::` line | Rejected: renders inside a `/** */` block, pollutes the whole-class listing, and relies on brittle directive-in-comment matching |
| Grow the `ResolveCtx`/codegen SPI to emit out-of-brace text | Rejected: over-broadens a public surface for a docs-only concern |

The overlay keeps the whole pipeline structured (JavaPoet in, JavaPoet out) and contains the change to the
one module that already owns the JavaPoet boundary.

### D3 — The overlay wins via `DuplicatesStrategy.EXCLUDE`, not an `exclude` filter

`percolate-javapoet` grows `src/main/java/com/palantir/javapoet/MethodSpec.java` (a vendored, minimally
patched copy incl. its nested `Builder`). It declares `package com.palantir.javapoet;`, so it compiles
against the `compileOnly` upstream jar with package-private access to `CodeWriter`/`CodeBlock`. The shadow
plugin's duplicate handling makes it shadow the upstream twin:

```
Duplicate detection runs BEFORE relocation, on the pre-relocation path.
Both entries collide at com/palantir/javapoet/MethodSpec.class at that moment.
Shadow processes "project files first", so under EXCLUDE the FIRST occurrence
(our overlay) survives and upstream's copy is dropped.
THEN relocation renames the survivor to io.github.joke.percolate.javapoet.MethodSpec,
rewriting our overlay's internal references (CodeWriter, …) along with it.
```

Config is scoped to the overlaid path only (rest of the jar keeps its default strategy):

```groovy
tasks.named('shadowJar') {
    relocate 'com.palantir.javapoet', 'io.github.joke.percolate.javapoet'
    filesMatching('com/palantir/javapoet/MethodSpec*.class') {   // catches MethodSpec$Builder
        duplicatesStrategy = DuplicatesStrategy.EXCLUDE
    }
}
```

Rejected alternative: a `dependencies { exclude 'com/palantir/javapoet/MethodSpec*.class' }` filter — works,
but reaches into the dependency artifact and is easier to get wrong than the first-wins ordering. The
"`duplicatesStrategy` takes precedence over transforming/relocating" caveat is *helpful* here (we want one
winner, not a transformer merge), so EXCLUDE is exactly right.

**Framing:** this promotes `percolate-javapoet` from *source-free relocation* to *relocation + selective
overlay* — a reusable seam. Today the overlay set is just `MethodSpec`; the pattern generalises to any future
JavaPoet behaviour tweak by dropping another `com/palantir/javapoet/Foo.java` in.

### D4 — Minimal `emit()` surface: a single `docTag` name

The overlay adds one nullable field (`docTag` name) + a `Builder.docTag(String)` setter, and ~4 lines at the
head and tail of `emit()`: when present, emit `// tag::<name>[]\n` before the annotations and
`// end::<name>[]\n` after the closing brace. No other behaviour changes; the diff against upstream stays
auditable. `BuildMethodBodies.renderMethod` stops calling `DocTags.wrap(body, …)` and instead sets
`methodSpecBuilder.docTag(methodName)` when `options.isDocTags()`. The `spi` `DocTags` utility (and its spec)
is retired — its CodeBlock body-wrap is no longer used.

### D5 — Doc-authoring fixes ride along

Whole-method emission still needs two authoring corrections to render cleanly:
- `indent=0` on every generated-impl `include::[tag=…]` across all feature pages → normalises the 2-space
  class-member indent to column 0.
- `tags=**` on `temporal-mapping.adoc`'s whole-class listing → keeps every line but strips the tag directive
  lines, so the full class renders without marker noise. (`compile-time-switches.adoc`'s *intentional*
  doctags-on/off demonstration keeps its plain `[]` include — it is deliberately showing the markers.)

### D6 — Coverage & invariants

- **The vendored overlay is treated as third-party code, exempt from the house gates.** `percolate-javapoet`
  applies spotless/PMD/error-prone/NullAway/jacoco via `java-base`; a verbatim 590-line upstream class fails
  PMD (God-class/complexity), jacoco coverage, and NullAway's `RequireExplicitNullMarking` under `-Werror`.
  So the module excludes `com/palantir/javapoet/**` from PMD, jacoco coverage, and error-prone (error-prone's
  `excludedPaths` also covers NullAway + `RequireExplicitNullMarking`, so no `package-info` is needed), and
  lets spotless format it. This keeps re-vendoring on a JavaPoet bump mechanical (no house-style rewrite of
  upstream code).
- The overlay's added behaviour is pinned by a focused Spock spec in `processor` (which already has the test
  harness and sees the *relocated* `io.github.joke.percolate.javapoet.MethodSpec` via `percolate-javapoet`'s
  shadowJar) — `MethodSpec.methodBuilder(…).docTag("m").build().toString()` brackets the whole method — and
  end-to-end by the docTags-on generation fixtures. `percolate-javapoet` itself stays source-lean (one
  vendored class, no test harness); the vendored class stays out of the coverage denominator.
- The `docTags`-off byte-identical invariant is re-asserted by the existing switches doctags-off fixture.
- `docTags`-on doc/e2e fixtures re-pin whole-method output; `processor` pitest floors held, never lowered
  ([[feedback_pitest_history_plugin]]); no jqwik ([[feedback_no_jqwik]]).
- Relocation swallow + ArchUnit guards from `relocate-javapoet-as-spi-api` still pass (still fully relocated,
  still zero upstream POM dep).

## Risks / Trade-offs

- **[Overlay drifts from upstream `MethodSpec` on a JavaPoet bump]** → JavaPoet is pinned at `0.16.0` and the
  palantir fork moves slowly; the overlay is a minimal, header-attributed diff. Re-vendor on any deliberate
  bump is a documented, contained chore, gated by the overlay spec + full `check`.
- **[Global `duplicatesStrategy` disturbs other shading]** → Scope EXCLUDE to `MethodSpec*` via
  `filesMatching`; the rest of `shadowJar` keeps its default. `percolate-javapoet` bundles only JavaPoet (no
  service files to merge), so blast radius is nil.
- **[Overlay silently fails to win → duplicate or upstream class shipped]** → The overlay spec asserts the
  emitted markers; a full `check` compiling `docTags`-on fixtures would fail loudly if upstream's unpatched
  `MethodSpec` shipped instead. Also assert the shaded jar carries exactly one `MethodSpec`.
- **[Whole-method region subtly shifts `docTags`-off output]** → The `docTag` field is null off-path and
  `emit()` is unchanged when null; re-verify byte-identical against the switches doctags-off fixture.
- **[Spotless/Guava-worker flake masks a real failure]** → Confirm via `git stash` on `main` before
  attributing ([[feedback_check_configcache_shipkit]]); run `./gradlew check --no-configuration-cache`.

## Migration Plan

1. **Overlay** — add `percolate-javapoet/src/main/java/com/palantir/javapoet/MethodSpec.java` (vendored +
   `docTag`), wire `filesMatching`/`EXCLUDE`, re-enable coverage, add the overlay spec.
2. **Rewire** — `BuildMethodBodies` (and the generate render leaf) from body-wrap to `methodSpecBuilder.docTag`;
   retire `spi` `DocTags` + its spec.
3. **Re-pin generation** — update `docTags`-on doc/e2e fixtures to whole-method output; re-verify off-path
   byte-identical.
4. **Doc authoring** — `indent=0` on all generated-impl tagged includes; `tags=**` on temporal's class listing.
5. **Gate + sync** — `./gradlew check` green, pitest floors held, relocation guards pass; sync deltas; archive.

Rollback: staged and behaviour-preserving; the overlay + rewire land together (step 1–2) but the doc-authoring
fixes (step 4) are independent and revertible on their own.

## Open Questions

- Whether to vendor the entire `MethodSpec.java` (simplest, larger sync surface) or trim it to the minimum
  compilable subset — settled at apply time; full-file vendoring is the default lean.
- Whether any non-temporal page also wants a whole-class listing (none today) — out of scope unless one
  appears.
