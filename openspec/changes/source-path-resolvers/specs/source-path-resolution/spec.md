## ADDED Requirements

### Requirement: PathSegmentResolver SPI

The `percolate-spi` module SHALL define a Java interface `io.github.joke.percolate.spi.PathSegmentResolver` with the following shape:

```java
public interface PathSegmentResolver {
    Optional<ResolvedSegment> resolve(
        TypeMirror parentType,
        String segment,
        ResolveCtx ctx);
}
```

Implementations SHALL return `Optional.of(ResolvedSegment)` when the resolver can produce a typed access for `segment` against a value of `parentType`, otherwise `Optional.empty()`. Implementations MUST NOT throw on a non-applicable input — they SHALL return `Optional.empty()` instead.

The SPI SHALL NOT import any class from `io.github.joke.percolate.processor.graph` or `io.github.joke.percolate.processor.stages.expand`. Resolver implementations describe the access; the caller (`SeedGraph`) is responsible for graph mutation.

The package `io.github.joke.percolate.spi` SHALL include `PathSegmentResolver` in its `@NullMarked`-marked surface.

#### Scenario: PathSegmentResolver with no match returns empty
- **WHEN** an implementor decides the resolver does not apply to the inputs
- **THEN** `resolve(...)` returns `Optional.empty()`
- **AND** does not throw

#### Scenario: PathSegmentResolver imports
- **WHEN** any concrete resolver implementation in `strategies-builtin/` is inspected
- **THEN** no import references `io.github.joke.percolate.processor.graph.*` or `io.github.joke.percolate.processor.stages.expand.*`

### Requirement: ResolvedSegment result type

The `percolate-spi` module SHALL define an immutable value type `io.github.joke.percolate.spi.ResolvedSegment` with three fields:

- `TypeMirror getReturnType()` — the type produced by the access
- `EdgeCodegen getCodegen()` — renders the access expression; receives one `IncomingValues` slot (the parent value) and a `VarNames` placeholder; the rendered expression evaluates to a value of `getReturnType()`
- `int getWeight()` — the strategy weight for the resulting REALISED edge

The type SHALL provide a single all-args constructor and accessor methods generated by Lombok's `@Value`. Equality is field-by-field.

#### Scenario: ResolvedSegment fields are immutable
- **WHEN** a `ResolvedSegment` is constructed via its all-args constructor
- **THEN** none of the three fields can be reassigned (no setters)

#### Scenario: ResolvedSegment carries the access codegen
- **WHEN** a resolver returns `Optional.of(new ResolvedSegment(returnType, codegen, weight))`
- **THEN** the caller can invoke `getCodegen().render(vars, inputs)` and receive a `CodeBlock` for the access expression

### Requirement: GetterPathResolver built-in

The `percolate-strategies-builtin` module SHALL ship `io.github.joke.percolate.spi.builtins.GetterPathResolver` implementing `PathSegmentResolver` and annotated `@AutoService(PathSegmentResolver.class)`.

`GetterPathResolver.resolve(parentType, segment, ctx)` SHALL:

1. Return `Optional.empty()` when `parentType.getKind() != TypeKind.DECLARED` or its element is not a `TypeElement`.
2. Probe for a JavaBean accessor method on `parentType`, in this order:
   - `get<Segment>()` where `<Segment>` is `segment` with its first character upper-cased (zero parameters, non-Object-class).
   - `is<Segment>()` where `<Segment>` is as above (zero parameters, return type is `boolean` / `java.lang.Boolean`, non-Object-class).
3. If neither matches, return `Optional.empty()`.
4. On match, return `Optional.of(new ResolvedSegment(method.getReturnType(), codegen, Weights.STEP))` where `codegen` renders `<slot-0>.<methodName>()`.

`GetterPathResolver` SHALL ignore methods declared on `java.lang.Object`.

#### Scenario: GetterPathResolver matches a JavaBean getter
- **WHEN** `GetterPathResolver.resolve(<Person>, "lastName", ctx)` is invoked and `Person` has `String getLastName()`
- **THEN** the returned `Optional` is non-empty
- **AND** `returnType` is `String`
- **AND** the codegen renders `<slot>.getLastName()`

#### Scenario: GetterPathResolver matches an `is` accessor for boolean
- **WHEN** `GetterPathResolver.resolve(<Person>, "active", ctx)` is invoked and `Person` has `boolean isActive()`
- **THEN** the returned `Optional` is non-empty
- **AND** the codegen renders `<slot>.isActive()`

#### Scenario: GetterPathResolver rejects parameterized overloads
- **WHEN** `GetterPathResolver.resolve(<Person>, "name", ctx)` is invoked and `Person` has both `String getName(String suffix)` and no zero-arg `getName()`
- **THEN** the returned `Optional` is empty

#### Scenario: GetterPathResolver ignores Object methods
- **WHEN** `GetterPathResolver.resolve(<Object>, "class", ctx)` is invoked
- **THEN** the returned `Optional` is empty even though `Object.getClass()` exists

#### Scenario: GetterPathResolver returns empty for non-declared parents
- **WHEN** `GetterPathResolver.resolve(<int[]>, "length", ctx)` is invoked (array, not declared)
- **THEN** the returned `Optional` is empty

### Requirement: RecordPathResolver built-in

The `percolate-strategies-builtin` module SHALL ship `io.github.joke.percolate.spi.builtins.RecordPathResolver` implementing `PathSegmentResolver` and annotated `@AutoService(PathSegmentResolver.class)`.

`RecordPathResolver.resolve(parentType, segment, ctx)` SHALL:

1. Return `Optional.empty()` when `parentType.getKind() != TypeKind.DECLARED` or its element is not a `TypeElement` of `ElementKind.RECORD`.
2. Probe for a zero-arg method named exactly `segment` on `parentType`.
3. If not matched, return `Optional.empty()`.
4. On match, return `Optional.of(new ResolvedSegment(method.getReturnType(), codegen, Weights.STEP))` where `codegen` renders `<slot-0>.<segment>()`.

#### Scenario: RecordPathResolver matches a canonical accessor
- **WHEN** `RecordPathResolver.resolve(<PointRecord>, "x", ctx)` is invoked and `PointRecord` is a `record PointRecord(int x, int y)`
- **THEN** the returned `Optional` is non-empty
- **AND** `returnType` is `int`
- **AND** the codegen renders `<slot>.x()`

#### Scenario: RecordPathResolver rejects non-record parents
- **WHEN** `RecordPathResolver.resolve(<Person>, "lastName", ctx)` is invoked and `Person` is a plain class (not a record)
- **THEN** the returned `Optional` is empty

### Requirement: FieldPathResolver built-in

The `percolate-strategies-builtin` module SHALL ship `io.github.joke.percolate.spi.builtins.FieldPathResolver` implementing `PathSegmentResolver` and annotated `@AutoService(PathSegmentResolver.class)`.

`FieldPathResolver.resolve(parentType, segment, ctx)` SHALL:

1. Return `Optional.empty()` when `parentType.getKind() != TypeKind.DECLARED` or its element is not a `TypeElement`.
2. Probe for a `VariableElement` on `parentType` whose `simpleName` equals `segment`, whose `ElementKind` is `FIELD`, and whose modifiers contain neither `PRIVATE` nor `STATIC`.
3. If not matched, return `Optional.empty()`.
4. On match, return `Optional.of(new ResolvedSegment(field.asType(), codegen, Weights.STEP))` where `codegen` renders `<slot-0>.<segment>` (a field read with no parentheses).

#### Scenario: FieldPathResolver matches a public field
- **WHEN** `FieldPathResolver.resolve(<Box>, "value", ctx)` is invoked and `Box` has `public String value`
- **THEN** the returned `Optional` is non-empty
- **AND** the codegen renders `<slot>.value`

#### Scenario: FieldPathResolver rejects private fields
- **WHEN** `FieldPathResolver.resolve(<Person>, "lastName", ctx)` is invoked and `Person` has `private String lastName` (Lombok `@Value`)
- **THEN** the returned `Optional` is empty

#### Scenario: FieldPathResolver rejects static fields
- **WHEN** `FieldPathResolver.resolve(<Constants>, "DEFAULT", ctx)` is invoked and `Constants` has `public static String DEFAULT`
- **THEN** the returned `Optional` is empty

### Requirement: Resolver registration via ServiceLoader

`PathSegmentResolver` implementations in `percolate-strategies-builtin` SHALL be discoverable via `java.util.ServiceLoader.load(PathSegmentResolver.class, classLoader)` through their `@AutoService(PathSegmentResolver.class)` registration. The processor SHALL collect them into an immutable `List<PathSegmentResolver>` sorted by `Class.getName()` (class-name ascending) and inject that list at the seed-time path-walking entry point.

#### Scenario: ServiceLoader discovers all three builtins
- **WHEN** `ServiceLoader.load(PathSegmentResolver.class, ProcessorModule.class.getClassLoader())` is invoked
- **THEN** the returned stream contains `FieldPathResolver`, `GetterPathResolver`, and `RecordPathResolver` (subset)

#### Scenario: Resolver list is sorted by class name
- **WHEN** the processor module's `pathSegmentResolvers()` provider is invoked
- **THEN** the returned list iterates in `Class.getName()` ascending order
- **AND** ties are impossible (class names are unique)

### Requirement: Seed-time path-walking algorithm

For every `MappingDirective` `d` on every method `M` whose `source` path has more than one segment (`segments = [s1, s2, ..., sk]` with `k >= 2`), `SeedGraph` SHALL walk the path from the parameter-root inward and produce one typed source node and one 1-slot `ExpansionGroup` per non-root segment `si` (for `i = 2..k`):

1. Begin with `parent = parameterRoot(s1)`, which is typed by the method's parameter declaration.
2. For each `i` from `2` to `k`:
   - Iterate the injected `pathSegmentResolvers` in class-name ascending order. For each resolver, call `resolve(parent.type, si, ctx)`. Take the first non-empty `Optional<ResolvedSegment>`; call it `rs`.
   - If no resolver matches, stop typing the chain for `d` (the remaining untyped chain stays as today; the directive's bridging edge in the next requirement falls back accordingly).
   - Allocate `typedNode = new Node(Optional.of(rs.returnType), SourceLocation([s1, ..., si]), MethodScope(M))`. Add it to the graph.
   - Allocate `realisedEdge = Edge.realised(parent, typedNode, rs.weight, rs.codegen, resolver.getClass().getName())`.
   - Allocate `markerEdge = Edge.marker(untypedSourceNodeFor([s1, ..., si]), typedNode, resolver.getClass().getName())`.
   - Register `group = ExpansionGroup.of(typedNode, List.of(parent), wrapAsGroupCodegen(rs.codegen), resolver.getClass().getName(), Set.of(realisedEdge), graph)` and call `graph.addGroup(group)`.
   - `parent = typedNode`.

The untyped seed source chain `parameterRoot → src[s1] → src[s1.s2] → ... → src[s1...sk]` (existing requirement *Directive-seeded source chains*) remains in the graph unchanged.

Within a single method scope, two directives sharing a prefix (e.g., both source paths beginning with `person.address`) SHALL reuse the typed node already registered for the shared prefix. `SeedGraph` SHALL key reuse on `(scope, full-path-segments)`. The second directive's walk SHALL NOT register a duplicate `ExpansionGroup` for the shared prefix segments.

Single-segment source paths (`segments.size() == 1`) SHALL NOT trigger path walking; they retain today's behaviour from the *Directive-seeded source chains* requirement.

#### Scenario: Two-segment source path produces one typed node and one GetterCall group
- **WHEN** `SeedGraph.apply(...)` is invoked for `@Map(target = "lastName", source = "person.lastName")` on a method `Human map(Person person)` with a registered `GetterPathResolver`, and `Person.getLastName()` returns `String`
- **THEN** the graph contains a typed node with `loc = SourceLocation(["person", "lastName"])`, `type = Optional.of(String)`, scope `MethodScope(<map>)`
- **AND** the graph contains a `REALISED` edge from `src[person]` (the parameter root) to the new typed node, weighted `Weights.STEP`, with codegen rendering `<input>.getLastName()`
- **AND** the graph contains a `MARKER` edge from `src[person.lastName] : ?` (the untyped seed leaf) to the new typed node
- **AND** `graph.groups()` contains exactly one new `ExpansionGroup` whose `root` is the typed node and whose single `slot` is `src[person]`

#### Scenario: Three-segment path produces two typed nodes and two groups in order
- **WHEN** `SeedGraph.apply(...)` is invoked for `@Map(target = "x", source = "person.address.street")` and `Person.getAddress()` returns `Address` and `Address.getStreet()` returns `String`
- **THEN** the graph contains typed nodes for `["person", "address"]` (`type = Address`) and `["person", "address", "street"]` (`type = String`)
- **AND** the graph contains two new `ExpansionGroup`s: one rooted at the `Address`-typed node with slot `src[person]`, one rooted at the `String`-typed node with slot at the `Address`-typed node
- **AND** the `REALISED` edges chain `src[person] → src[person.address] (typed) → src[person.address.street] (typed)`

#### Scenario: No resolver matches; the typed chain stops at that segment
- **WHEN** `SeedGraph.apply(...)` is invoked for `@Map(target = "x", source = "person.weirdSegment")` and no registered resolver matches the segment `weirdSegment` on `Person`
- **THEN** no typed node is allocated for `["person", "weirdSegment"]`
- **AND** no new `ExpansionGroup` is registered for `weirdSegment`
- **AND** the existing untyped seed chain remains in the graph
- **AND** the directive's bridging edge (next requirement) originates from the untyped seed leaf

#### Scenario: Two directives sharing a prefix reuse the typed node
- **WHEN** `SeedGraph.apply(...)` is invoked for two directives `@Map(source = "person.address.street")` and `@Map(source = "person.address.city")` on the same method, where `Person.getAddress()` returns `Address`
- **THEN** exactly one typed node exists for `["person", "address"]`
- **AND** exactly one `ExpansionGroup` is registered for the `Address`-segment (slot `src[person]`)
- **AND** the typed nodes for `["person", "address", "street"]` and `["person", "address", "city"]` are both registered separately, each as the root of its own 1-slot group

#### Scenario: Single-segment source path is unaffected
- **WHEN** `SeedGraph.apply(...)` is invoked for `@Map(target = "x", source = "person")` with no further segments
- **THEN** no resolver is invoked
- **AND** no typed source node is allocated for `["person"]` beyond the parameter root
- **AND** the existing `Directive-seeded source chains` behaviour applies

### Requirement: Resolver priority determinism

When multiple `PathSegmentResolver` implementations match the same `(parentType, segment)` input, the seed-time algorithm SHALL select the first match in class-name ascending order. Selection SHALL be byte-stable across multiple JVM runs given the same registered resolver set.

Today's class-name ordering yields `FieldPathResolver` < `GetterPathResolver` < `RecordPathResolver`. A field access wins over a same-named getter on the same parent type. Changing this default requires changing `ProcessorModule.pathSegmentResolvers()`'s comparator, not the SPI surface.

#### Scenario: Field wins when a public field and a same-named getter coexist
- **WHEN** the seed-time algorithm invokes resolvers for `(<Bean>, "value", ctx)` where `Bean` has both `public String value` and `String getValue()`
- **THEN** the first matching resolver is `FieldPathResolver`
- **AND** the registered group's codegen renders `<slot>.value` (the field read)

#### Scenario: Resolver order is deterministic across runs
- **WHEN** the seed-time algorithm runs twice against the same mapper input in two JVM invocations
- **THEN** the resulting typed source chains, REALISED edges, and registered groups are byte-identical (modulo node identityHashCode)

### Requirement: Existing GetterRead-as-Bridge is not affected

This change SHALL NOT modify the behaviour, registration, or contract of `io.github.joke.percolate.spi.builtins.GetterRead` (the `Bridge`-as-getter implementation). `GetterRead` continues to register via `@AutoService(Bridge.class)` and continues to fire during `ExpandGroupsPhase` for cases not covered by seed-time resolution. Removal of `GetterRead` is explicitly deferred to a follow-up change.

#### Scenario: GetterRead-as-Bridge remains registered
- **WHEN** `ServiceLoader.load(Bridge.class, ProcessorModule.class.getClassLoader())` is invoked after this change is applied
- **THEN** the returned stream contains `GetterRead` (subset assertion)

#### Scenario: GetterRead-as-Bridge unit spec is unchanged
- **WHEN** `GetterReadSpec.groovy` is inspected after this change is applied
- **THEN** every scenario from before this change still passes
